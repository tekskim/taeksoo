3. 비즈니스 로직만 추가

03

왜 전환해야 하는가

5가지 핵심 장점

핵심 장점

🎯 디자인-코드 동기화 문제 해결

컴포넌트가 곧 디자인이므로 "해석" 자체가 불필요

근거: 1,425줄 AI Rules가 간격·색상·타이포 등 모든 규칙을 정의, AI가 정확히 따름

⚡ 개발 속도 10x 향상

페이지당 2-4시간 → 10-20분으로 단축

근거: 315개 페이지가 동일 패턴으로 구현됨. Phase 2에서 263개 일괄 마이그레이션 성공

🔒 일관성 보장

어떤 개발자가 만들어도 동일한 결과

근거: 282/315 페이지(89.5%)가 동일한 PageShell 패턴 사용 중

🤖 AI 친화적 아키텍처

Rules = AI의 디자인 가이드. 명확한 API로 높은 코드 생성 정확도

근거: 263개 페이지 PageShell 마이그레이션을 AI가 오류 없이 수행

디자이너 리소스 효율화

현재 (비효율)

  • 반복적인 CRUD 화면을 매번 그림
  • 리스트/상세 페이지 80%가 동일 패턴
  • 디자이너의 시간이 "그리기"에 소모

제안 (효율)

  • CRUD 화면은 TDS 패턴으로 자동 생성
  • 디자이너는 새로운 패턴 설계에 집중
  • UX 리서치와 시스템 디자인에 역량 투입
디자이너 역할: "화면 그리기" → "시스템 디자인 + UX 리서치"

알고 있는 리스크, 준비된 대응

리스크 및 완화 방안

🎨 디자인 탐색/실험 제약 리스크: 중

신규 레이아웃/인터랙션을 코드 기반으로 탐색하기 어려움

→ 신규 패턴 = Figma, 반복 화면 = TDS (하이브리드)

👀 리뷰 채널 부재 리스크: 중

PM/경영진의 Figma 리뷰 프로세스가 사라짐

→ Storybook 배포 + Chromatic CI를 리뷰 채널로 활용

초기 투자 필요 해소됨

PageShell, EmptyState, ErrorState 등 구현 필요

→ Phase 1·2에서 6개 컴포넌트 구현 + 282개 페이지 적용 완료

엣지 케이스 처리 리스크: 낮음

대시보드, 모니터링 등 특수 화면

→ 80/20 원칙: CRUD(80%) = TDS, 특수(20%) = 기존 방식. 실측: 미적용 10.5%만

개발자에게 의미하는 것

✅ 더 이상 추측하지 않아도 됩니다

"이 간격이 8px인지 12px인지" Figma에서 추측하는 작업이 사라집니다. 모든 규칙이 코드에 있습니다.

✅ 학습 비용이 줄어듭니다

컴포넌트 API만 알면 됩니다. 1,425줄 Rules가 AI를 가이드하므로 자연어로 화면을 요청하면 됩니다.

✅ 비즈니스 로직에 집중할 수 있습니다

현재 커스텀 코드 30% → UI 코드 대부분이 DS 컴포넌트로 대체되어 핵심 로직에 집중 가능합니다.

✅ 유연성도 확보되어 있습니다

className, children 패턴으로 커스텀 escape hatch가 있습니다. 경직된 시스템이 아닙니다.
