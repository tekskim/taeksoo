# 디자인-개발 핸드오버 프로세스 제안

## 현재 상황

### 디자인팀

- Cursor 기반 코드로 모든 앱 화면을 디자인 (TDS)
- 자체 컴포넌트 라이브러리 구축 완료 (62개 컴포넌트)
- Figma 파일이 아닌 실제 동작하는 React 앱으로 디자인 결과물 보유
- Compute 앱 전 화면을 Figma로 디자인해본 경험이 있으나, 수정사항 반영과 최신화에 어려움을 겪음

### 개발팀

- Figma MCP 기반 워크플로우를 구축해놓음
- Figma 파일에서 `get_design_context`로 디자인 정보를 읽고, 자체 컴포넌트 라이브러리(`@thaki/shared`)로 코드를 생성
- 현재 규칙상 Figma 파일이 없으면 UI 구현 작업을 시작하지 않음

### 두 팀의 컴포넌트 라이브러리 차이

| 항목 | 디자인팀 (TDS) | 개발팀 (@thaki/shared) |
|---|---|---|
| 프레임워크 | React | React |
| 아이콘 | Tabler Icons | Phosphor Icons |
| 스타일링 | Tailwind + CSS 변수 | CSS Modules + 디자인 토큰 |
| 컴포넌트 예시 | `<Select>`, `<Modal>`, `<EmptyState>` | `<Dropdown>`, `<Overlay.Template>`, `<EmptyUI>` |

컴포넌트 이름과 Props API가 상당히 다르지만, 기능적으로 대응 관계가 명확함.

## 문제 정의

### 1. Figma 변환 품질 문제

디자인팀의 코드 기반 결과물을 Figma로 변환하려면 html.to.design 플러그인을 사용해야 하는데, 변환 결과물은:

- 레이어 이름이 `Frame 1268`, `Group 5` 등 의미 없는 이름
- 컴포넌트 인스턴스가 아닌 일반 프레임으로 생성
- 인터랙티브 상태(hover, disabled, error 등) 미캡처

이 상태에서 개발팀의 AI가 Figma를 읽으면, 시각적으로 추측해서 컴포넌트를 매핑해야 하므로 정확한 코드 생성이 보장되지 않음.

### 2. 규모의 문제

전체 변환 대상: **443개 프레임** (AI Platform, Agent 제외)

| 카테고리 | Home | List | Detail(탭별) | Create(스텝별) | 기타 | 소계 |
|---|---|---|---|---|---|---|
| Entry/Settings | 9 | - | - | - | - | 9 |
| Compute | 1 | 20 | 53 | 30 | 2 | 106 |
| Compute Admin | 1 | 24 | 65 | 24 | 2 | 116 |
| Container | 2 | 22 | 58 | 67 | 41 | 190 |
| IAM | 1 | 12 | 13 | 10 | - | 36 |
| Storage | 1 | 7 | 15 | 3 | - | 26 |
| **합계** | | | | | | **483** |

> YAML 에디터(39개)와 Console(3개)은 동일 템플릿이므로 실질 **443개 프레임**.

### 3. 유지보수의 문제 (가장 큰 문제)

443개 Figma 프레임을 만드는 것도 문제이지만, **이후 유지보수가 더 큰 문제**:

- 디자인 수정 시: 코드에서는 1줄 수정이면 끝나지만, Figma에서는 수백 개 프레임을 하나씩 찾아서 수정해야 함
- 컴포넌트 변경 시: Button 스타일 하나 바뀌면 Button이 사용된 모든 페이지 프레임을 업데이트해야 함
- 이미 Compute 앱에서 이 문제를 경험함 — 전 화면 Figma 작업 후 수정사항 반영과 최신화에 어려움

**코드 → Figma → 코드** 경로는 두 번의 변환에서 정보 손실이 발생하고, 양방향 동기화 비용이 지속적으로 발생함.

## 제안: 코드 기반 핸드오버 + 선택적 Figma

### 기본 원칙

```
기존 패턴의 페이지 구현 → 코드 기반 핸드오버 (Figma 불필요)
신규 컴포넌트 개발     → Figma 비주얼 스펙 제공 (필요할 때만)
```

### 코드 기반 핸드오버 (기본)

디자인팀에서 제공하는 것:

| 전달물 | 내용 | 유지보수 |
|---|---|---|
| **배포된 TDS 앱** | 전체 페이지 비주얼 레퍼런스 (실제 동작) | 코드 수정 시 자동 반영 |
| **배포된 Storybook** | 62개 컴포넌트의 모든 상태, Props, 인터랙션 | 코드 수정 시 자동 반영 |
| **컴포넌트 매핑 규칙 파일** | TDS 컴포넌트 → @thaki/shared 컴포넌트 매핑 테이블 | 라이브러리 변경 시 업데이트 |

컴포넌트 매핑 규칙 파일은 개발팀 레포의 `.cursor/rules/`에 추가하여, 개발팀의 AI가 TDS 디자인을 볼 때 자동으로 `@thaki/shared` 컴포넌트로 변환하도록 함.

매핑 예시:

| TDS | @thaki/shared | Props 변환 |
|---|---|---|
| `<Button variant="primary">` | `<Button variant="primary">` | `appearance` prop 추가 |
| `<Select options={...}>` | `<Dropdown.Select>` + `<Dropdown.Option>` | 구조 변경 |
| `<Modal isOpen title>` | `<Overlay.Template type="modal">` | API 전면 변경 |
| `<Table data={...}>` | `<Table rows={...}>` | `data` → `rows` |
| `<Badge variant="info">` | `<Badge theme="blu">` | `variant` → `theme` (축약형) |
| Tabler `<IconPlus>` | `<CreateIcon>` from @thaki/shared | 아이콘 라이브러리 변경 |

### Figma 제공 (필요할 때만)

Figma 비주얼 스펙이 필요한 경우:

| 케이스 | 예시 | Figma 제공 내용 |
|---|---|---|
| 개발팀에 대응 컴포넌트가 없을 때 | Popover, MetricCard 등 Gap 컴포넌트 | 모든 상태의 비주얼 스펙 |
| 기존 컴포넌트의 디자인 변경이 필요할 때 | Button에 새 variant 추가 등 | 변경 사항의 비주얼 스펙 |
| 완전히 새로운 페이지 패턴이 필요할 때 | 기존 List/Detail/Create 패턴에 해당하지 않는 페이지 | 해당 페이지 비주얼 스펙 |

Figma가 **불필요한** 경우:

- 기존 패턴(List, Detail, Create)과 동일한 구조의 새 페이지
- 이미 매핑 가능한 컴포넌트만으로 구성된 페이지
- 레이아웃만 다르고 컴포넌트는 동일한 페이지

### 유지보수 비용 비교

| | Figma 443페이지 유지 | 코드 기반 핸드오버 |
|---|---|---|
| **초기 작업** | 443 프레임 변환 + 후처리 | 매핑 규칙 작성 + 배포 |
| **디자인 수정 시** | Figma 프레임 수동 업데이트 | 코드 수정 → 배포 자동 반영 |
| **컴포넌트 변경 시** | 전체 프레임에서 수동 교체 | 매핑 규칙 1줄 수정 |
| **신규 페이지 추가 시** | 새 프레임 생성 + 후처리 | 코드 작성 → 배포 자동 반영 |
| **지속 비용** | 높음 (동기화 필수) | 낮음 (코드가 곧 최신 상태) |

## 실행 계획

### Phase 1: 파일럿 테스트 (1-2일)

검증 목적으로 대표 페이지 2개를 두 가지 방식으로 비교 테스트:

- **방식 A**: html.to.design으로 Figma 변환 → 개발팀 `figma-to-code` 실행
- **방식 B**: TDS 비주얼 레퍼런스 + 매핑 규칙 → 개발팀 코드 생성

비교 항목: 코드 품질, 소요 시간, @thaki/shared 컴포넌트 정확도

### Phase 2: 매핑 규칙 파일 작성 (2-3일)

- TDS 62개 컴포넌트 → @thaki/shared 68개 컴포넌트 매핑
- Props 변환 규칙, 아이콘 매핑, 디자인 토큰 매핑 포함
- 개발팀 레포에 `.cursor/rules/10-frontend/tds-component-mapping.mdc`로 추가

### Phase 3: 배포 (1일)

- TDS 앱 공개 URL 배포
- TDS Storybook 공개 URL 배포

### Phase 4: 개발팀 워크플로우 조정 (협의)

개발팀의 기존 Figma 필수 규칙을 이원화:

```
기존: Figma 없으면 작업 불가
변경: Figma 또는 TDS 비주얼 레퍼런스 + 매핑 규칙
```

### Phase 5: Gap 컴포넌트 Figma 제공 (필요 시)

개발팀에 대응 컴포넌트가 없는 것들만 Figma 비주얼 스펙으로 제공 (약 6개 컴포넌트).

---

## 개발팀 협의 필요 사항

1. **Figma 필수 규칙 완화 가능 여부** — TDS 비주얼 레퍼런스를 Figma 대안으로 허용할 수 있는지
2. **파일럿 테스트 참여** — 대표 페이지 2개로 두 방식을 비교 테스트
3. **매핑 규칙 파일 수용** — `.cursor/rules/`에 매핑 규칙 추가 동의
4. **@thaki/shared 컴포넌트 목록 공유** — 정확한 매핑을 위해 최신 컴포넌트 API 문서 필요
